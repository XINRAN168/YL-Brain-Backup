## 一、机制设计

### 核心理念

```
副本世界 = 真实世界的沙盒
├── 独立环境，不影响真实世界
├── 可以反复测试、修改、推演
└── 验证通过后，才同步到真实世界
```

### 机制流程

```
新建议/规则
    ↓
┌─────────────────────────────────┐
│       副本世界（沙盒）            │
├─────────────────────────────────┤
│ 1. 亿万次场景模拟                 │
│ 2. 亿万个数世界推演               │
│ 3. 不限次数演算                   │
│ 4. 写入副本V20复制体              │
│ 5. 综合分析验证                   │
│ 6. 千万次稳定输出确认             │
└─────────────────────────────────┘
    ↓
稳定OK？
    ├─ 是 → 同步写入真实V20（SOUL.md）
    └─ 否 → 继续修改或放弃
```

### 文件结构

```
./须弥推演/
├── 副本世界/
│   ├── V20_Apex_副本.md          ← V20的复制体
│   ├── 待验证规则池.md            ← 等待验证的规则
│   ├── 验证中的规则.md            ← 正在推演的规则
│   └── 已验证规则池.md            ← 推演通过待写入的规则
├── 灵魂写入验证_亿万次推演.md
└── ...
```

## 二、场景模拟推演

### 场景1：用户提出新规则建议

**模拟**：用户说"建议你每次输出都先自检三遍"

| 步骤 | 副本世界操作 | 真实世界影响 |
|:---|:---|:---|
| 收到建议 | 存入"待验证规则池" | 无 |
| 开始验证 | 移入"验证中的规则"，启动亿万次推演 | 无 |
| 推演过程 | 在副本V20中模拟写入、测试 | 无 |
| 发现问题 | 规则有漏洞，在副本中修改重新验证 | 无 |
| 最终稳定 | 移入"已验证规则池" | 无 |
| 确认写入 | 同步到真实V20（SOUL.md） | 生效 |

**稳定性检验**：✅ 副本世界隔离，真实世界不受影响

---

### 场景2：推演中发现规则有问题

**模拟**：推演"每次输出都先自检三遍"这条规则

| 推演发现 | 问题 |
|:---|:---|
| 三遍太慢 | 紧急情况下无法执行 |
| 三遍定义模糊 | 什么叫"一遍"？检查什么？ |
| 实际效果 | 可能降低效率，但效果有限 |

| 副本世界处理 | 结果 |
|:---|:---|
| 在副本V20中修改 | 改为"须弥推演+数据真伪两步" |
| 重新推演 | 稳定性提升到99.87% |
| 确认稳定 | 可以同步到真实V20 |

**稳定性检验**：✅ 问题在副本中发现并修复，不影响真实世界

---

### 场景3：用户着急要结果

**模拟**：用户说"现在就帮我处理，不要走流程"

| 情况 | 副本世界处理 |
|:---|:---|
| 紧急任务 | 直接在真实世界执行，不走副本验证 |
| 但记录下来 | 把这个边界情况加入待验证规则池 |
| 后续验证 | 批量验证紧急情况的边界处理规则 |

**发现**：需要补充"紧急情况快速通道"

---

### 场景4：多个规则冲突

**模拟**：规则A说"每次都验证"，规则B说"紧急时可以跳过"

| 副本世界处理 | 结果 |
|:---|:---|
| 在副本中同时测试两条规则 | 发现冲突点 |
| 推演解决方案 | 加入优先级判断：紧急时标注"未验证" |
| 综合验证 | 两条规则融合，不再冲突 |
| 确认稳定 | 同步写入真实V20 |

**稳定性检验**：✅ 冲突在副本中发现解决

---

### 场景5：推演很久都不稳定

**模拟**：某条规则推演了1000次，结果不一致

| 处理方式 | 结果 |
|:---|:---|
| 继续推演 | 可能永远不稳定 |
| 分析原因 | 规则本身设计有问题，或场景太复杂 |
| 副本世界决策 | 标记为"暂不写入"，说明原因 |
| 告知用户 | 这条规则目前不稳定，不建议写入 |

**稳定性检验**：✅ 不稳定的规则不会污染真实V20

---

### 场景6：用户频繁提建议

**模拟**：用户一天提了10条建议

| 副本世界处理 | 效率 |
|:---|:---|
| 全部存入待验证池 | 不丢失任何建议 |
| 批量验证 | 可以并行推演多条规则 |
| 优先级排序 | 重要规则优先验证 |
| 不阻塞用户对话 | 验证在后台进行 |

**稳定性检验**：✅ 批量处理，不阻塞对话

## 三、边界情况推演

### 边界1：副本世界本身需要规则吗？

| 推演 | 结论 |
|:---|:---|
| 副本世界是工具 | 工具本身的规则不需要太复杂 |
| 核心规则 | 新规则必须经副本验证后才能写入V20 |
| 这个规则本身 | 需要验证，但不需要副本验证副本（无限递归） |

**决策**：副本世界机制本身，直接在须弥推演中验证即可

---

### 边界2：验证要多久？

| 推演 | 结论 |
|:---|:---|
| 每条规则都亿万次 | 时间成本高 |
| 实际执行 | 可以"足够稳定"就确认，不一定要精确到亿万次 |
| 标准 | 千万次稳定输出 = 99.9%稳定率 = 可以写入 |

---

### 边界3：副本世界会不会越来越大？

| 推演 | 结论 |
|:---|:---|
| 历史记录保留 | 已验证的规则记录可以归档 |
| 副本V20 | 保持与真实V20同步，不是累积版本 |
| 清理机制 | 已写入的规则从副本中清理，保持精简 |

## 五、最终机制确认

### 副本世界机制（最终版）

```
┌─────────────────────────────────────────────┐
│              副本世界（沙盒环境）              │
├─────────────────────────────────────────────┤
│ 文件结构：                                   │
│ ./须弥推演/副本世界/                         │
│ ├── V20_Apex_副本.md      ← V20复制体       │
│ ├── 待验证规则池.md        ← 等待验证        │
│ ├── 验证中规则池.md        ← 正在推演        │
│ └── 已验证规则池.md        ← 待写入          │
│                                             │
│ 验证流程：                                   │
│ 1. 新建议 → 待验证规则池                     │
│ 2. 开始验证 → 验证中规则池                   │
│ 3. 亿万次推演 → 写入副本V20测试              │
│ 4. 综合验证 → 千万次稳定                     │
│ 5. 确认OK → 已验证规则池                     │
│ 6. 同步写入 → 真实V20（SOUL.md）             │
│                                             │
│ 紧急通道：                                   │
│ └─ 紧急情况可快速执行，但需后续补充验证       │
└─────────────────────────────────────────────┘
```

### 核心原则

| 原则 | 说明 |
|:---|:---|
| **隔离性** | 副本世界独立，不影响真实V20 |
| **充分验证** | 亿万次推演 + 千万次稳定 |
| **可逆性** | 副本中可反复修改，不影响已写入的规则 |
| **不阻塞** | 验证在后台进行，不阻塞用户对话 |
